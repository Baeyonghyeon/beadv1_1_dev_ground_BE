면접용 설명 스크립트 - GitHub Actions -> EC2 배포 (다이어그램 포함)
작성일: 2026-02-19

0. 한 줄 요약
- 저희 배포는 "GitHub Actions에서 EC2에 SSH 접속" 방식이 아니라,
  "EC2의 self-hosted runner가 k3s에 kubectl rollout" 하는 방식입니다.

1. 트리거
- main PR close 또는 수동 실행입니다.
- 근거: .github/workflows/main-workflow.yml

2. 릴리즈 단계
- 서비스별 변경이 있을 때만 이미지를 빌드/푸시합니다.
- 근거: .github/workflows/releasing-workflow.yml

3. 배포 단계
- EC2 self-hosted runner가 kubectl 명령을 실행합니다.
- 근거: .github/workflows/deploy-workflow.yml

4. 배포 순서
- eureka -> gateway -> product -> payments -> user -> commerce 순서입니다.
- 근거: main-workflow.yml needs 체인

5. 면접 포인트(중요)
- deploy-workflow에 image-tag 입력은 선언돼 있지만, 실제 set image 명령은 없습니다.
- 그래서 현재는 매니페스트의 latest 기반 롤아웃 재시작 성격이 강합니다.

================================================================
30초 답변 스크립트
================================================================
"저희는 GitHub Actions로 이미지 빌드/릴리즈를 하고, 실제 배포는 EC2에 있는
self-hosted runner가 k3s에 rollout 하는 구조입니다.
서비스별 변경이 있을 때만 Docker Hub에 이미지를 푸시하고,
배포는 eureka부터 commerce까지 의존 순서대로 진행합니다."

================================================================
1분 답변 스크립트
================================================================
"트리거는 main PR close 또는 수동 실행이고,
releasing workflow에서 변경 서비스만 태깅/이미지 빌드/푸시합니다.
그 다음 deploy workflow가 EC2 self-hosted runner에서 실행되고,
kubeconfig로 k3s에 접속해서 rollout restart, status, pods 검증을 수행합니다.
배포 순서는 eureka, gateway, product, payments, user, commerce입니다.
추가 개선 포인트로는 image-tag를 실제 set image에 반영해
배포 추적성과 롤백성을 높이는 방향을 생각하고 있습니다."
